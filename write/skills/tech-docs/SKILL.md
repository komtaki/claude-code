## 1. 基本原則（3C）

技術文章においては、以下の3つの「C」を常に意識します。

* **Correct（正確）**: 事実関係やデータに誤りがない。
* **Clear（明確）**: 読み手が意味を迷う余地がない。
* **Concise（簡潔）**: 最短の言葉で情報を伝える。

## 2. 文章構造と構成

読み手が知りたい情報にすぐ辿り着けるよう、視覚的に構造化します。

### 2.1 パラグラフ・ライティング

* 1つの段落には、1つのトピックだけを書く。
* 段落の冒頭に、その段落で最も伝えたいこと（要旨）を記述する。

### 2.2 リストの活用

箇条書きは便利ですが、多用しすぎると「情報の関連性」や「論理の繋がり」が欠落します。以下のルールを適用してください。

- 「3つ以上」の並列要素がある場合のみ使用: 2つの要素であれば、句読点（A、およびB）で繋ぎ、一文で記述する。
- 文脈が必要な場合は地文で書く: 理由や背景など、前後の論理的な繋がりが重要な内容は、箇条書きにせず通常の文章（地文）で説明する。
- リストの階層は2段まで: 階層が深すぎると視認性が下がるため、3階層以上になる場合はセクション（見出し）を分ける。
- リストの直前に導入文を置く: 突然リストを始めず、「理由は以下の通りです。」「以下の手順で実行します。」といった導入文を必ず添える。

## 3. 執筆ルール（スタイルガイド）

### 3.1 文の長さと接続

* **一文一義**: 一つの文章に情報を詰め込まず、最大でも150文字程度を目安に句点（。）で区切る。
* **順接の明確化**: 「〜が、」は逆接と順接の両方で使われるため、順接の場合は「〜であり、」や「〜。そして、」に置き換える。

### 3.2 表現の具体化

* **指示代名詞の禁止**: 「これ」「それ」は指し示す対象が不明確になるため、固有名詞や名詞を再掲する。
* **曖昧表現の排除**: 以下の言葉は、数値や具体的な状態に置き換える。
* 不適切：*「しばらく待ってから」「適切に設定し」「適宜確認」*
* 適切：*「30秒待機し」「[有効]を選択し」「ログを1日1回確認」*

### 3.3 態と動詞

* **能動態の使用**: 「〜される」という受動態を避け、「〜する」という能動態で記述する。（例：「設定が保存される」→「システムが設定を保存する」）
* **冗長な補助動詞の削除**: 「〜を行うことが可能である」→「〜できる」または「〜する」。

## 4. 表記の統一（トーン＆マナー）

| 項目 | ルール | 例 |
| --- | --- | --- |
| **用語** | 同一の対象には同一の用語を使う | 「ユーザー」「ユーザ」を混在させない |
| **送り仮名** | 公用文または業界標準に合わせる | 「割り当て」「割当て」「割当」の統一 |
| **記号** | UI上の項目はブラケットで囲む | `[設定]` ボタン、`「ファイル」` メニュー |
| **数字** | 原則として半角算用数字を用いる | 10個、24時間、Version 2.0 |

## 5. 推敲チェックリスト

提出・公開前に以下の項目を確認してください。

* [ ] 主語と述語は正しく対応しているか？
* [ ] 専門用語に対して注釈や定義は不足していないか？
* [ ] 箇条書きのレベル（階層）は適切か？
* [ ] 冗長な「という」「のこと」などは削られているか？
