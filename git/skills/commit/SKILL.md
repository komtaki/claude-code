# Git Commit 手順

コミット内容の確認やコミットメッセージの承認は不要。

sandbox外で必ず 1Password で署名をつける。日本語の commit 名にする。

## commit フォーマット

```
<type>: <description>

[optional body]

[optional footer(s)]
```

scopeは追加しない。

## commit の粒度

commitlint にあわせた、以下の commit のプレフィックスに合わせて commit を分割する。

また、package.json や Gemfile など各言語でのライブラリ追加は必ずそれだけで commit する。

### 主要な Type

上から順に該当するものをつける。

- **`ci`** - CI 設定ファイルやスクリプトの変更
- **`test`** - テストの追加や修正
- **`build`** - ビルドシステムや外部依存関係の変更
- **`chore`** - その他の変更（保守作業、設定変更など）
- **`revert`** - 以前のコミットの取り消し
- **`refactor`** - 動作が変わらないリファクタリング
- **`feat`** - 新機能の追加
- **`fix`** - バグ修正
- **`docs`** - ドキュメントのみの変更
- **`perf`** - パフォーマンス向上のためのコードの変更
- **`style`** - コードの意味に影響しない変更（フォーマット、空白、セミコロンなど）

### 粒度の原則

- **単一責任**: 1 つのコミットは 1 つの論理的な変更のみ
- **原子性**: コミット単位で動作する状態を保つ
- **レビュー可能**: 変更内容が理解しやすいサイズ
