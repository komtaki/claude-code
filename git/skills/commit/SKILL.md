# Git Commit 手順

sandbox外で必ず 1Password で署名をつける。日本語の commit 名にする。

## commit フォーマット

```
<type>: <description>

[optional body]

[optional footer(s)]
```

scopeは追加しない。

## commit の粒度

commitlint にあわせた、以下の commit のプレフィックスに合わせて commit を分割する。

また、package.json や Gemfile など各言語でのライブラリ追加は必ずそれだけで commit する。

### 主要な Type

上から順に該当するものをつける。

- **`ci`** - CI 設定ファイルやスクリプトの変更
- **`test`** - テストの追加や修正
- **`build`** - ビルドシステムや外部依存関係の変更
- **`chore`** - その他の変更（保守作業、設定変更など）
- **`revert`** - 以前のコミットの取り消し
- **`refactor`** - 動作が変わらないリファクタリング
- **`feat`** - 新機能の追加
- **`fix`** - バグ修正
- **`docs`** - ドキュメントのみの変更
- **`perf`** - パフォーマンス向上のためのコードの変更
- **`style`** - コードの意味に影響しない変更（フォーマット、空白、セミコロンなど）

## 実践的な例

### 良い例

```bash
# 新機能追加
feat: ユーザープロフィールページを追加
feat: ファイルアップロード機能を実装
feat: 検索フィルターコンポーネントを追加

# バグ修正
fix: コンポーネントアンマウント時のメモリリークを解決
fix: ネットワークタイムアウトエラーを適切に処理
fix: メールアドレスのバリデーションロジックを修正

# リファクタリング
refactor: 共通ボタンスタイルを共有コンポーネントに抽出
refactor: データ処理ロジックを簡素化
refactor: データベースクエリのパフォーマンスを最適化

# ドキュメント
docs: APIドキュメントを更新
docs: 開発環境セットアップガイドを追加

# テスト
test: 認証機能のユニットテストを追加
test: APIエンドポイントの統合テストを追加

# 設定・依存関係
chore: 依存関係を最新バージョンに更新
build: バンドルサイズ設定を最適化
ci: 自動テストパイプラインを追加
```

### 避けるべき例

```bash
# 曖昧すぎる
fix: バグ修正
feat: 新機能
update: 変更

# 複数の変更を含む
feat: ユーザーエンドポイントを追加し認証バグを修正
```

## コミット粒度のガイドライン

### 適切な粒度

- **単一責任**: 1 つのコミットは 1 つの論理的な変更のみ
- **原子性**: コミット単位で動作する状態を保つ
- **レビュー可能**: 変更内容が理解しやすいサイズ

### 推奨される分割例

```bash
# 大きすぎる
feat: チャート、フィルター、エクスポート機能を含む完全なダッシュボードを実装

# 適切に分割
feat: 基本的なダッシュボードデータ構造を追加
feat: チャートコンポーネントを実装
feat: 日付範囲フィルター機能を追加
feat: CSVエクスポート機能を実装
test: テストケースの追加
```
